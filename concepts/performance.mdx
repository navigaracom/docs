Navigara measures engineering output as three scores per commit: **grow**, **maintenance**, and **waste**. Each score reflects both the type of work and its complexity, computed from a combination of AI classification and deterministic code analysis. Every measurement happens at the individual file change level, not at the commit level — a single commit touching five files produces five independent scores, each with its own work type, complexity, and engagement.

## How it works

Every commit goes through two stages.

**Stage 1: AI analysis.** Navigara reads the commit message, the full diff, and the surrounding code context. For each changed file, the model classifies the type of work (feature, fix, refactor), identifies the changed symbols (functions, classes, endpoints), and, for bug fixes, traces the issue back to the originating commit, recording the original author and timestamp. Results are stored in a knowledge graph connecting commits, files, symbols, and issues.

**Stage 2: Mechanical scoring.** Deterministic algorithms compute complexity and engagement scores over the same files. No LLM is involved at this stage. Scores are reproducible and consistent across runs.

The AI determines what kind of work was done. The algorithms determine how hard the code is. The Performance score combines both.

## What gets analyzed

Navigara analyzes every source file modified in a commit. Not all file changes are scored — before analysis begins, the following are automatically filtered out:

- **Generated code** — Protocol Buffer outputs (`.pb.go`, `_grpc.pb.go`, `.pb.ts`), GraphQL codegen (`.graphql.ts`), OpenAPI specs, and other machine-generated files (`*_generated.go`, `*.gen.go`, `zz_generated.*`).
- **Dependency lockfiles** — `go.sum`, `package-lock.json`, `yarn.lock`, `pnpm-lock.yaml`, `Cargo.lock`, `Gemfile.lock`, and similar.
- **Build artifacts** — `dist/`, `build/`, `.next/`, `vendor/`, `node_modules/`, and bundled outputs with content hashes (e.g. `main-YHGF2JUB.js`).
- **Minified files** — detected by content heuristics when average line length exceeds 300 characters.
- **Binary and media files** — images, fonts, PDFs, archives, compiled binaries.

These files are excluded from both the AI analysis and mechanical scoring. If you see a commit with fewer scored files than total files changed, the difference is filtered files. The knowledge graph records which files were filtered and why.

## Work types

The AI classifies each file change into one of three work types:

| Type | Labels | What it captures |
|------|--------|------------------|
| **Grow** | `feat` | New features and capabilities |
| **Maintenance** | `chore`, `refactor` | Upkeep, cleanup, and restructuring |
| **Waste** | `fix` | Bug fixes — work that corrects previous output |

Classification happens per file, not per commit. A single commit can contain feature work in one file and a bug fix in another — each is scored independently.

## How the score is calculated

Each file change produces a performance score from two mechanical signals: complexity and engagement.

**Complexity** combines two established measures. [Halstead complexity](https://en.wikipedia.org/wiki/Halstead_complexity_measures) (adapted) analyzes operators and operands to measure the vocabulary richness of the code. The formula is adapted to work fairly across languages and paradigms — a React component blending JSX, JavaScript, and CSS-in-JS would score artificially high under raw Halstead, so the adapted formula accounts for this. [Cyclomatic complexity](https://en.wikipedia.org/wiki/Cyclomatic_complexity) measures the number of independent execution paths. Every `if`, `for`, `while`, `case`, and logical operator (`&&`, `||`) adds a path. Together, Halstead measures how much you need to read, while cyclomatic complexity measures how many paths you need to reason about.

**Engagement** estimates how much of the surrounding context a developer needed to understand to make the change:

$$
\text{engagement} = \sqrt{\frac{\text{lines changed}}{\text{total lines}}}
$$

Changing a few lines in a large file means you likely only needed to understand a small slice of it. Changing most of a file means you had to hold the whole thing in your head. The square root provides diminishing returns — changing 25% of a file gives roughly 50% engagement.

**The final score** for each file change is:

$$
\text{performance} = \frac{\text{complexity} \times \text{engagement}}{100}
$$

File-level scores are summed per work type, producing three numbers — grow, maintenance, and waste — for any contributor, team, or repository over any time window.

## Waste multiplier

When the AI identifies a file change as a bug fix, it traces the deleted or modified lines back to the commit that introduced the bug — recording the original author, timestamp, and file churn history. This feeds into a waste multiplier that adjusts the score based on how costly the bug was:

| Factor | What it measures | Effect |
|--------|-----------------|--------|
| **Time since origin** | How old was the buggy code? | Older bugs are more expensive — they have been in production longer |
| **Author** | Did the same person fix it? | Fixes by a different developer cost more (context transfer) |
| **File churn** | How often has this file changed recently? | High-churn files signal systemic quality issues |

A trivial self-fix on code written the same day has a multiplier near 1.0x. An old bug fixed by a different developer in a high-churn file can reach roughly 5.0x.

<Tip>
A healthy codebase typically shows high grow, moderate maintenance, and low waste. A spike in waste may signal quality issues worth investigating.
</Tip>

## Architecture multiplier

The base performance score measures how complex the code itself is. But not all code operates in isolation — a change to a billing endpoint that coordinates frontend forms, backend validation, a payment gateway, and database transactions is harder than a change of equal cyclomatic complexity in a standalone utility.

Navigara captures this through the **architecture multiplier**, powered by a dedicated [architect agent](/concepts/architecture) that runs per repository.

### How the architect agent works

When [architecture analysis](/concepts/architecture) is enabled, a multi-phase agent explores the codebase and builds a structured map:

1. **Feature discovery** — identifies distinct features (e.g. `auth`, `billing`, `notifications`) from code structure, naming conventions, and module boundaries.
2. **Vertical stack mapping** — classifies each feature's files into layers: frontend, backend, and data.
3. **Dependency mapping** — traces dependencies and dependents between features within the repository.

The result is a feature graph where each node is a feature with known layers and edges represent coupling between features. See [Architecture](/concepts/architecture) for details on how features are discovered and matched to commits.

### How the multiplier is calculated

For each feature, the architecture multiplier reflects the complexity of its input/output surface:

| Factor | What it measures | Effect |
|--------|-----------------|--------|
| **Vertical span** | How many layers does the feature touch? | A feature spanning frontend, backend, and data is harder to change than one confined to a single layer |
| **Dependency count** | How many other features depend on this one? | More dependents means more constraints on changes |
| **Dependent count** | How many features does this one consume? | More dependencies means more context to hold |

A self-contained backend utility with no dependencies gets a multiplier near **1.0x**. A feature that spans all three layers and sits at the center of the dependency graph can reach **3.0x** or higher.

When a commit touches files belonging to a feature, the file-level performance scores are multiplied by that feature's architecture multiplier:

$$
\text{adjusted performance} = \text{base performance} \times \text{architecture multiplier}
$$

### Repository mesh

Individual repositories don't exist in a vacuum. In a microservices or monorepo-adjacent setup, features in one repository often depend on features in another — a frontend app consumes a backend API, which in turn depends on a shared auth library.

When multiple repositories are connected to the same organization, Navigara builds a **repository mesh** — a cross-repository dependency graph where edges represent inter-repo feature dependencies. These emerge automatically when the architect agent detects:

- **API contracts** — a frontend feature importing types generated from a backend proto or OpenAPI spec.
- **Shared libraries** — multiple repositories depending on the same internal package.
- **Event flows** — a producer feature in one repo and a consumer feature in another.

The mesh adds an additional layer to the architecture multiplier. A feature that is a dependency for features in other repositories carries more weight than one with only intra-repo dependents, because changes to it have a wider blast radius.

| Scope | Multiplier range | Example |
|-------|-----------------|---------|
| **Intra-repo only** | 1.0x – 3.0x | A feature used only within its own repository |
| **Cross-repo dependency** | 1.5x – 5.0x | A shared auth library consumed by three services |

<Tip>
The architecture multiplier rewards engineers working on high-leverage, structurally complex parts of the system. Two commits with identical diffs will score differently if one touches a central feature with cross-repo dependents and the other touches an isolated utility.
</Tip>

## Supporting roles

Senior engineers often spend significant time on code reviews, architectural design, mentoring, and technical decision-making — work that does not produce commits. The Performance score does not capture these contributions directly. We are aware of this gap and plan to introduce metrics for non-code contributions in the future.

We believe this is best addressed by measuring teams rather than individuals. An engineer in a supporting role raises the output quality of everyone around them — fewer bugs, better architecture, faster onboarding. That impact shows up in the team's aggregate performance even when the individual's commit-based score is low. This is why we recommend using team-level views as the primary lens. Individual scores are useful for understanding work distribution, but they do not tell the full story when supporting roles are involved.

## KTLO mode

By default, Navigara shows the full three-bucket breakdown: Grow / Maintenance / Waste. If you prefer a simpler view, you can switch to **Grow / KTLO** mode in **Settings > General**.

KTLO (Keep The Lights On) combines Maintenance and Waste into a single category. This is useful when communicating with stakeholders outside of engineering — instead of explaining three categories, you get a clean split between new value (Grow) and everything else (KTLO).

| Mode | Categories | Best for |
|------|-----------|----------|
| **Grow / Maintenance / Waste** | 3 buckets | Engineering teams that want full visibility into where effort goes |
| **Grow / KTLO** | 2 buckets | Executive reporting and cross-team communication |

<Note>
Switching metrics mode only changes how the data is displayed. The underlying analysis stays the same and you can switch back at any time without losing data.
</Note>
