title: "Performance"

The Performance score measures **what your team builds and how complex the work is**. It combines AI understanding of each file with mechanical code analysis to produce three scores per commit — **grow**, **maintenance**, and **waste** — that capture both the nature and the difficulty of the work.

## How it works: AI + mechanical analysis

Navigara uses a two-stage pipeline for every commit:

1. **AI analysis (knowledge graph)** — An LLM reads the commit message, the full diff, and the surrounding code context. For each changed file, it classifies the type of work (feature, fix, refactor), identifies the changed symbols (functions, classes, endpoints), and for bug fixes, root-causes the issue — tracing it back to the originating commit and author. These results are stored as a **knowledge graph** that connects commits, files, symbols, and issues into a queryable structure.

2. **Mechanical scoring** — Deterministic algorithms then run over the same files to compute complexity and engagement scores. These are purely code-based — no LLM involved — so they're reproducible and consistent.

The AI decides _what kind of work_ was done and _why_. The algorithms measure _how hard_ the code is. The performance score is the product of both.

## Work types

The AI reads each file's diff and classifies it into one of three work types:

| Type | Labels | What it captures |
|------|--------|------------------|
| **Grow** | `feat` | New features and capabilities |
| **Maintenance** | `chore`, `refactor` | Upkeep, cleanup, and restructuring |
| **Waste** | `fix` | Bug fixes — work that corrects previous output |

This classification happens **per file**, not per commit — a single commit can contain both feature work and bug fixes across different files.

## How the Performance score is calculated

Each file's performance score combines two mechanical signals:

### Complexity

Navigara computes a complexity score for each file by combining two established measures:

**Halstead complexity** (adapted) — analyzes operators (keywords, control flow, symbols) and operands (identifiers, numbers, strings) to measure the vocabulary richness of the code. Navigara uses a formula [inspired by Halstead](https://en.wikipedia.org/wiki/Halstead_complexity_measures) but adapted to work fairly across languages and paradigms. For example, a React component blending JSX, JavaScript, and CSS-in-JS would score artificially high under raw Halstead because it looks like three languages merged together — the adapted formula accounts for this.

**Cyclomatic complexity** — measures the number of independent execution paths through the code, based on [McCabe's Cyclomatic Complexity](https://en.wikipedia.org/wiki/Cyclomatic_complexity). Every `if`, `for`, `while`, `case`, and logical operator (`&&`, `||`) adds a path. Code with many branches is harder to understand and test than linear code, even if it has fewer lines.

Together, these capture two distinct dimensions of difficulty: Halstead measures _how much_ you need to read, while cyclomatic complexity measures _how many paths_ you need to reason about.

### Engagement

Engagement estimates **how much of the surrounding context a developer needed to understand** to implement the change:

$$
\text{engagement} = \sqrt{\frac{\text{lines changed}}{\text{total lines}}}
$$

Changing a few lines in a large file means you likely only needed to understand a small slice of it. Changing most of a file means you had to hold the whole thing in your head. The square root provides diminishing returns — changing 25% of a file gives ~50% engagement, reflecting that understanding grows faster than the raw line ratio.

### Putting it together

$$
\text{performance} = \frac{\text{complexity} \times \text{engagement}}{100}
$$

File-level scores are summed per work type, giving you three numbers — **grow**, **maintenance**, and **waste** — for any contributor, team, or repository over any time window.

## Waste multiplier

When the AI identifies a file change as a bug fix, it root-causes the issue by tracing the deleted or modified lines back to the commit that introduced them — identifying _who_ wrote the original code, _when_, and _how often_ the file has been churning. This context feeds into a **waste multiplier** that increases the score based on how costly the bug was to the organization:

| Factor | What it measures | Effect |
|--------|-----------------|--------|
| **Time since origin** | How old was the buggy code? | Older bugs are more expensive — they've been in production longer |
| **Author** | Did the same person fix it? | Fixes by a different developer cost more (context transfer) |
| **File churn** | How often has this file been changed recently? | High-churn files signal systemic quality issues |

A trivial self-fix on code written the same day has a multiplier near **1.0×**. An old bug fixed by a different developer in a high-churn file can reach **~5.0×**.

<Tip>
A healthy codebase typically shows high **grow**, moderate **maintenance**, and low **waste**. A spike in waste may signal quality issues worth investigating.
</Tip>

## KTLO mode

By default, Navigara shows the full three-bucket breakdown: **Grow / Maintenance / Waste**. If you prefer a simpler view, you can switch to **Grow / KTLO** mode.

**KTLO** (Keep The Lights On) combines Maintenance and Waste into a single category. This is useful when communicating with stakeholders outside of engineering — instead of explaining three categories, you get a clean split between _new value_ (Grow) and _everything else_ (KTLO).

### How to enable it

Go to **Settings → General** and select the **Grow / KTLO** radio option under the metrics mode section. The change takes effect immediately across all dashboards and reports for your organization.

| Mode | Categories | Best for |
|------|-----------|----------|
| **Grow / Maintenance / Waste** | 3 buckets | Engineering teams that want full visibility into where effort goes |
| **Grow / KTLO** | 2 buckets | Executive reporting and cross-team communication |

<Note>
Switching metrics mode only changes how the data is displayed — the underlying analysis stays the same. You can switch back at any time without losing data.
</Note>
